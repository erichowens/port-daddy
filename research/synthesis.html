Executive Summary

Three research streams converged this sprint. The findings are more bullish than expected.

Multi-agent systems (2026): The industry has converged on a two-layer stack — MCP for agent-infrastructure, A2A for agent-agent. Both are under the Linux Foundation. The 10 canonical archetypes (Orchestrator, Planner, Executor, Critic…) provide a shared vocabulary for swarm design. The dominant failure mode: “Bag of Agents” — N agents with overlapping responsibilities and no archetype assignment. Port Daddy’s session + salvage system naturally enforces archetype clarity: agents declare purpose at registration, and dead agents leave a readable audit trail.

Local LLMs: Running 10 agents for an hour costs $0.07 with Ollama (electricity only). The key discovery: don’t spawn the claude CLI as a subprocess — 7× token overhead from MCP re-init. Use @anthropic-ai/sdk directly. lib/spawner.ts is ~200 lines. Port Daddy already has everything else.

Docs: Tutorials 06 and 08 are stubs (13% quality score). Zero tutorials have screenshots. The API reference doesn’t exist — it’s a link to GitHub. All fixable this sprint.

Geeked Out The A2A Agent Card pattern (/.well-known/agent.json) is the single most exciting finding. When spawned agents serve Agent Cards, Port Daddy becomes a capability registry — other agents ask “who can do X?” and get a live list. PD goes from port manager to service mesh for agents. One sprint away.

Protocol Landscape

Four protocols compete for the agent communication stack. Two are dominant; two are niche.

MCP

Model Context Protocol

Anthropic → Linux Foundation. Agent ↔ Infrastructure. Pull-based: agents declare needs, server provides. The de-facto standard for tool use.

Port Daddy uses this today. Our MCP server is how Claude agents connect to PD.

A2A

Agent-to-Agent Protocol

Google → Linux Foundation. Agent ↔ Agent. Push-based task delegation. Agent Cards at /.well-known/agent.json for capability discovery.

PD should serve Agent Cards for spawned agents. Right long-term design.

ACP

Agent Comm. Protocol

IBM. Lightweight A2A alternative, fewer primitives, REST-first. For simple agent pipelines.

Watch list only. No action needed for PD v3.x.

Agent Protocol

Agent Protocol

HuggingFace / Community. Standardized client-side interface. Defines task lifecycle: create → step → complete.

Useful reference for pd spawn task lifecycle design.

Critical Finding A2A’s binary message parts mean agents can exchange embedding vectors directly — not just text. This enables semantic memory passing without re-encoding. PD notes already accept arbitrary content; future tooling can attach embeddings enabling semantic search across agent memory.

10 Agent Archetypes

The “Bag of Agents” anti-pattern: N agents with overlapping responsibilities, no archetype assignment, constant coordination failures. Solution: name archetypes upfront.

01
Orchestrator
Receives goals, decomposes, delegates to workers. Never does direct work. The “conductor” of the swarm.
02
Planner
Converts goals into step-by-step task graphs. Produces DAGs. Distinct from Orchestrator: plans, doesn’t direct.
03
Executor
Runs atomic tasks. No planning, pure execution. Highly parallelizable. The “worker.” Should be stateless between tasks.
04
Critic
Reviews outputs against quality criteria. Adversarial by design. Essential for preventing “yes-man” loops in homogeneous swarms.
05
Evaluator
Scores outputs with rubrics. Produces numbers, not prose. Enables automatic quality gates; routes back to Executors if score is too low.
06
Synthesizer
Combines outputs from multiple workers into coherent results. Resolves contradictions. Creates the final artifact humans see.
07
Retriever
Queries stores (docs, code, vector memory) to provide context. Manages RAG pipelines. Pre-fetches relevant context before Executor starts.
08
Memory Keeper
Maintains persistent state across sessions and restarts. Port Daddy itself fills this role for session + port state.
09
Mediator
Resolves conflicts between agents: resource contention, scheduling disagreements. PD’s lock system is a Mediator primitive.
10
Monitor
Observability. Tracks agent health, detects failures, triggers salvage. PD’s resurrection queue is Monitor infrastructure. pd watch is the Monitor agent.
Cross-Reference pd begin --identity myapp:orchestrator --purpose "Plan auth refactor" now has a semantic home in this taxonomy. The identity’s stack component maps directly to archetype: myapp:executor, myapp:critic, myapp:synthesizer. Consider documenting this in the next tutorial sprint.

Swarm Patterns

Ant Colony Optimization

Biological pheromone trails map to shared message queues. Agents follow high-quality paths:

Pheromone trail simulation (CSS-only) 
# Agent publishes quality score ("pheromone") after completing a task
pd pub task-solutions '{"task":"auth","quality":0.92,"approach":"JWT"}'

# Next agents read scores before picking approaches
pd sub task-solutions   # pick highest-quality past approach

# Low-quality approaches "evaporate" when channel TTL expires

Stigmergy (Indirect Coordination)

Agents coordinate without direct messaging by modifying shared state. Port Daddy’s file claims + pub/sub are already a stigmergy substrate: Agent A claims src/auth.ts → Agent B detects claim and picks a different file. No direct agent-to-agent calls needed.

Geeked Out Heterogeneous swarms outperform homogeneous ones. A swarm of 4 specialized models (Planner + 2 Executors + Critic) beats 4 identical GPT-4o instances on complex tasks, even when individual specialized models are individually weaker. This validates pd spawn’s multi-backend design: mix Ollama (cheap, fast) + Claude Haiku (reasoning) + Aider (git-native coding).

Always-On Agents

The ambient agent pattern: a persistent process that subscribes to events and dispatches agents when conditions match — without human prompting.

Harbors / RBAC for Agents

Current PD problem: any agent can read/write any channel, claim any file. Fine for solo development; breaks down with untrusted agent code or multi-tenant environments.

Design Sketch (v4.0)

# Create a permission namespace
pd harbor create myapp:security-review \
  --cap "code:read,security:scan,notes:write" \
  --expires 2h

# Agents enter the harbor (scoped JWT tokens issued)
pd harbor enter myapp:security-review --agent reviewer-1

# Operations outside capabilities return 403
pd harbor leave myapp:security-review --agent reviewer-1

# List all harbors
pd harbors
Not This Sprint Harbors require JWT-based capability tokens + enforcement middleware. Needs careful crypto design. Plan for v4.0. Meanwhile, pd spawn’s identity scoping (project:stack:context) provides soft harbor semantics without enforcement.

pd spawn Design

The minimal implementation: lib/spawner.ts (~200 lines) + routes + CLI. PD already has everything else.

CLI Syntax

# Ollama: local, free, fast iteration
pd spawn --backend ollama --model llama3.2:8b \
  --identity myapp:coder --purpose "Refactor auth" \
  -- "Fix login bug in src/auth.ts"

# Claude: best reasoning, direct SDK (NOT CLI subprocess)
pd spawn --backend claude --model claude-haiku-4-5-20251001 \
  --identity myapp:reviewer \
  -- "Review PR #42 for security issues"

# Aider: git-native coding agent, makes commits
pd spawn --backend aider --model gemini/gemini-2.0-flash-exp \
  -- src/auth.ts src/middleware.ts

# Management
pd spawned                     # list running agents
pd spawn kill <agentId>        # stop a spawned agent

Auto-Wired Coordination

Every spawned agent automatically: registers with PD, calls pd.begin(), sends heartbeat every 30s, writes each LLM response as a note, calls pd.done() on exit. If killed unexpectedly: marked stale → resurrection queue after 10 min.

Cross-Reference A spawned agent that crashes is automatically recoverable: another agent can pd salvage claim <id> and continue from its notes. This is the coordination flywheel: every new feature strengthens every existing feature.
