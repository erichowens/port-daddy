DNS Resolution

Access your dev services by hostname instead of port numbers. Map myapp-api.local to your running service with a single command.

What is Port Daddy DNS?

Port Daddy maintains a local DNS registry that maps semantic service identities to hostnames. Instead of remembering that your API is on localhost:3100 and your frontend on localhost:5173, you register each service with a hostname and access them by name.

The registry lives in the Port Daddy daemon and can optionally write to your operating system's /etc/hosts file. Once that's done, any tool on your machine -- curl, your browser, other services -- can reach myapp-api.local without knowing the port number.

This is especially useful in multi-service and multi-agent workflows where services depend on each other by address. Agents can hardcode http://myapp-api.local instead of passing port numbers through environment variables.

Prerequisites

You'll need Port Daddy installed and the daemon running. If you haven't done this yet, see the Getting Started tutorial.

# Check that the daemon is running
$ pd status
Port Daddy daemon running on localhost:9876
Uptime: 2h 14m

If the daemon isn't running, start it:

$ pd start
Port Daddy daemon started on localhost:9876

Register Your First DNS Record

The simplest way to register a DNS record is to add --dns when claiming a port. Port Daddy derives a hostname from the service identity automatically.

$ pd claim myapp:api --dns
Port 3100 assigned to myapp:api
DNS: myapp-api.local

The derived hostname replaces colons with hyphens and appends .local. So myapp:api becomes myapp-api.local.

You can also register a DNS record separately from port claiming, and supply a custom hostname:

$ pd dns register myapp:api --hostname myapp-api.local
DNS record registered: myapp-api.local -> 127.0.0.1:3100 (myapp:api)

Verify the record was stored:

$ pd dns list
HOSTNAME               ADDRESS          SERVICE
myapp-api.local        127.0.0.1:3100   myapp:api
Note: DNS records in Port Daddy's registry are separate from OS-level resolution. The record is stored in the daemon, but your browser and curl won't use it until you run pd dns sync (covered below).

Register Multiple Services

For a typical web application with a backend API, frontend, and background job runner, you'd register all three at once:

$ pd claim myapp:api --dns
Port 3100 assigned to myapp:api
DNS: myapp-api.local

$ pd claim myapp:frontend --dns
Port 3101 assigned to myapp:frontend
DNS: myapp-frontend.local

$ pd claim myapp:jobs --dns
Port 3102 assigned to myapp:jobs
DNS: myapp-jobs.local

Now check the registry:

$ pd dns list
HOSTNAME                  ADDRESS          SERVICE
myapp-api.local           127.0.0.1:3100   myapp:api
myapp-frontend.local      127.0.0.1:3101   myapp:frontend
myapp-jobs.local          127.0.0.1:3102   myapp:jobs

All three services are now registered. The port assignments are stable -- if another agent or process calls pd claim myapp:api, it gets back port 3100 again. The DNS entry stays consistent.

Enable OS-Level Resolution with /etc/hosts

Port Daddy's internal registry isn't visible to your OS by default. To make hostnames like myapp-api.local work in curl, your browser, and other services, you need to write them to /etc/hosts.

Port Daddy manages a clearly delimited section of /etc/hosts and never touches the rest of the file.

Step 1: Set up the managed section

This adds Port Daddy's managed block to /etc/hosts. It requires sudo because /etc/hosts is root-owned.

$ sudo pd dns setup
Backup created: /etc/hosts.portdaddy-backup-1741654320
Port Daddy section added to /etc/hosts

Step 2: Sync DNS records to /etc/hosts

$ pd dns sync
Synced 3 record(s) to /etc/hosts
  myapp-api.local
  myapp-frontend.local
  myapp-jobs.local

This command does not require sudo -- Port Daddy only writes within its own managed section.

Step 3: Verify

$ pd dns status
DNS registry: 3 record(s)
/etc/hosts: synced (3 entries)
Last sync: just now

What /etc/hosts looks like before and after

Before:

127.0.0.1   localhost
255.255.255.255  broadcasthost
::1         localhost

After:

127.0.0.1   localhost
255.255.255.255  broadcasthost
::1         localhost
# === PORT DADDY DNS (managed, do not edit) ===
127.0.0.1   myapp-api.local
127.0.0.1   myapp-frontend.local
127.0.0.1   myapp-jobs.local
# === END PORT DADDY DNS ===
Do not edit the managed section by hand. Any changes between the Port Daddy markers will be overwritten the next time pd dns sync runs.

Test it with curl

$ curl http://myapp-api.local:3100/health
{"status":"ok","service":"myapp:api"}
It works. Your service is now accessible by hostname from any tool on your machine. Other services in the same project can call http://myapp-api.local:3100 instead of hard-coding a port number.

Check DNS Status

Two commands give you visibility into the DNS state:

$ pd dns status
DNS registry: 3 record(s)
/etc/hosts: synced (3 entries)
Last sync: 4 minutes ago

$ pd dns resolver
Resolver mode: /etc/hosts
Managed section: present
Records in registry: 3
Records synced to /etc/hosts: 3
Drift: none

pd dns resolver tells you whether the registry and /etc/hosts are in sync. If you see drift (records in the registry not in /etc/hosts), run pd dns sync to reconcile.

SDK Usage

All DNS operations are available in the JavaScript SDK for use in agent code and scripts:

const pd = new PortDaddy();

// Claim a port with DNS registration in one call
const { port } = await pd.claim('myapp:api');
await pd.dnsRegister('myapp:api', { hostname: 'myapp-api.local' });

// List all registered DNS records
const records = await pd.dnsList();
// [{ hostname: 'myapp-api.local', address: '127.0.0.1', port: 3100, service: 'myapp:api' }]

// Set up the /etc/hosts managed section (requires sudo context)
await pd.dnsSetup();

// Write registry to /etc/hosts
await pd.dnsSync();

// Check current status
const status = await pd.dnsStatus();
// { records: 3, synced: 3, lastSync: '2026-03-10T...' }

// Remove a record
await pd.dnsRemove('myapp:api');

The SDK is useful when you want DNS registration to be part of your service startup script. Register the hostname, claim the port, then signal other agents that the service is ready.

Troubleshooting

Permission error when running pd dns setup

$ pd dns setup
Error: permission denied writing to /etc/hosts
Hint: run with sudo: sudo pd dns setup

/etc/hosts is owned by root on macOS and Linux. The setup command needs sudo once to create the managed section. After that, pd dns sync can update the section without sudo because Port Daddy manages write access to its own block.

Hostname conflict -- record already exists

$ pd dns register myapp:api --hostname myapp-api.local
Error: hostname myapp-api.local is already registered (by myapp:api-v2)

Remove the conflicting record first, then re-register:

$ pd dns remove myapp-api.local
Record removed: myapp-api.local

$ pd dns register myapp:api --hostname myapp-api.local
DNS record registered: myapp-api.local -> 127.0.0.1:3100 (myapp:api)

Hostname not resolving in browser after sync

Browsers cache DNS responses. After running pd dns sync, if a hostname still doesn't resolve in your browser:

  1. Verify the entry is in /etc/hosts: grep -A 10 "PORT DADDY" /etc/hosts
  2. Restart your browser completely (quit and reopen, not just a new tab)
  3. On macOS, flush the DNS cache: sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder

Drift between registry and /etc/hosts

$ pd dns resolver
Drift detected: 2 record(s) in registry not in /etc/hosts

$ pd dns sync
Synced 2 new record(s) to /etc/hosts

Drift happens when you register new DNS records after the last sync. Running pd dns sync always brings them back into agreement.

Verify /etc/hosts entries directly

$ grep -A 20 "PORT DADDY" /etc/hosts
# === PORT DADDY DNS (managed, do not edit) ===
127.0.0.1   myapp-api.local
127.0.0.1   myapp-frontend.local
127.0.0.1   myapp-jobs.local
# === END PORT DADDY DNS ===

Cleanup

When you're done with a project or want to remove DNS records:

# Remove one specific record from the registry
$ pd dns remove myapp:api
Record removed: myapp-api.local (myapp:api)

# Sync the change to /etc/hosts
$ pd dns sync
Synced 2 record(s) to /etc/hosts (1 removed)

# Or remove everything: all records + the managed /etc/hosts section
$ sudo pd dns teardown
Backup created: /etc/hosts.portdaddy-backup-1741654987
Port Daddy section removed from /etc/hosts
3 DNS records cleared from registry

A backup of /etc/hosts is created automatically before any modifications. The backup filename includes a timestamp so you can restore a specific version if needed:

# Restore from backup if something went wrong
$ sudo cp /etc/hosts.portdaddy-backup-1741654987 /etc/hosts
