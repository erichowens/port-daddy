Session Phases & Integration

Track agent progress and coordinate service startup with structured phases and integration signals. Like PR status labels, but for running agent work.

What are session phases?

Sessions have a lifecycle. Phases let agents communicate their current state -- to other agents, to humans watching the dashboard, and to the salvage system if something goes wrong.

Think of phases like status labels on a pull request: In Review, Changes Requested, Merged. They don't enforce anything. They're a shared signal that answers the question "where are you in your work right now?" without requiring a dedicated communication channel.

Phases also feed the salvage system. When an agent dies and another agent picks up its work, the phase tells the rescuer exactly where the previous agent stopped. An agent with phase testing died while running tests -- the rescuer knows to look at test output, not at the implementation.

The 6 Phases

Sessions can be in one of six phases. They flow roughly left to right, with abandoned as an escape hatch from anywhere:

PhaseWhen to use it
planning Designing the approach, writing specs, deciding which files to touch
in_progress Actively writing code, making changes, executing the plan
testing Running the test suite, fixing failures, verifying behavior
reviewing Code review in progress, waiting for human approval or another agent's critique
completed Work merged or definitively done -- set automatically by pd done
abandoned Work stopped due to a blocker -- triggers salvage so another agent can continue

Starting a Session with a Phase

Pass --phase to pd begin to start directly in a specific phase. The default is planning if you omit it.

# Start in planning phase (default)
$ pd begin --identity myapp:feature-auth --purpose "Add JWT auth" --phase planning
Agent registered: agent-a1b2c3
Session started: sess-d4e5f6 [planning]
Context saved to .portdaddy/current.json
# Jump straight to in_progress if you already know what you're doing
$ pd begin --identity myapp:bugfix-timeout --purpose "Fix connection timeout" --phase in_progress
Agent registered: agent-b2c3d4
Session started: sess-e5f6a7 [in_progress]
Context saved to .portdaddy/current.json
Identity matters for salvage. The --identity flag tells Port Daddy which project and stack you're working on. When another agent in the same project runs pd salvage, they'll see your session and know what you were doing.

Advancing Through Phases

Update the phase as your work progresses. This keeps other agents and humans informed of exactly where you are.

# Done planning, starting to code
$ pd session update sess-d4e5f6 --phase in_progress
Session sess-d4e5f6 updated: planning -> in_progress

# Coding done, running tests
$ pd session update sess-d4e5f6 --phase testing
Session sess-d4e5f6 updated: in_progress -> testing

# Tests pass, requesting review
$ pd session update sess-d4e5f6 --phase reviewing
Session sess-d4e5f6 updated: testing -> reviewing

# Merged! pd done marks it completed automatically
$ pd done "Auth merged, all tests green"
Session ended: sess-d4e5f6 [completed]
Agent unregistered
Context cleaned up

You can also update the phase via the HTTP API directly -- useful when automating from CI or a shell script:

$ curl -X PUT http://localhost:9876/sessions/sess-d4e5f6 \
    -H "Content-Type: application/json" \
    -d '{"phase": "testing"}'
{"id":"sess-d4e5f6","phase":"testing","updatedAt":"2026-03-10T..."}

Abandoning a session

If you hit a blocker and can't continue, mark the session as abandoned. This places it in the salvage queue so another agent can pick it up with full context.

$ pd session update sess-d4e5f6 --phase abandoned
Session sess-d4e5f6 marked as abandoned
Entering salvage queue...

# Another agent sees it
$ pd salvage --project myapp
1 session(s) needing salvage in myapp:*

Session: sess-d4e5f6
Agent:   agent-a1b2c3 (dead)
Phase:   abandoned
Purpose: Add JWT auth
Notes:   4 note(s)
Files:   src/auth.ts, src/middleware.ts
Abandoned is not the same as completed. Use abandoned when you've stopped due to a blocker and want another agent to continue. Use completed (via pd done) when the work is actually done.

Integration Signals

Integration signals solve the startup coordination problem: "Is service X ready yet?" Instead of polling /health endpoints in a loop or hardcoding sleep 5 in your startup scripts, services declare when they're ready and waiters block until that signal arrives.

The basic pattern

# auth-service signals it's ready
$ pd integration ready auth-service
Signal broadcast: auth-service is ready

# api-service blocks until auth-service is ready
$ pd integration wait auth-service
Waiting for auth-service...
auth-service is ready!

If auth-service is already ready when wait is called, it returns immediately:

$ pd integration wait auth-service
auth-service is already ready. Proceeding immediately.

Coordinating a multi-service startup

Here's a real pattern for a three-service app where the API depends on auth and the frontend depends on the API:

# auth-service/start.sh
node src/auth-server.js &
# Wait for the server to bind and pass health check
until curl -sf http://localhost:3100/health >/dev/null; do sleep 0.5; done
pd integration ready auth-service

# api-service/start.sh
pd integration wait auth-service  # blocks until auth is up
node src/api-server.js &
until curl -sf http://localhost:3101/health >/dev/null; do sleep 0.5; done
pd integration ready api-service

# frontend/start.sh
pd integration wait api-service   # blocks until API is up
npm run dev

Each service declares its own readiness after confirming its health endpoint passes. Downstream services wait on the upstream signal. No sleep timers, no race conditions.

Using integration with sessions

Integration signals work well alongside session phases. An agent can advance to in_progress and then block on a dependency before doing its work:

$ pd begin --identity myapp:api --purpose "Build API endpoints" --phase planning
$ pd session update sess-d4e5f6 --phase in_progress

# Wait for the database migration agent to finish
$ pd integration wait myapp-db-migrated
Waiting for myapp-db-migrated...
myapp-db-migrated is ready!

# Now do the actual API work
$ pd session update sess-d4e5f6 --phase testing
$ pd done "API endpoints complete"

Several commands let you inspect active and past sessions:

$ pd sessions

SDK Usage

All phase and integration operations are available in the JavaScript SDK:

const pd = new PortDaddy();

// Start session with a phase
const { sessionId } = await pd.begin({
  identity: 'myapp:feature-auth',
  purpose: 'Add JWT authentication',
  phase: 'planning',
});

// Advance through phases
await pd.updateSession(sessionId, { phase: 'in_progress' });
await pd.updateSession(sessionId, { phase: 'testing' });

// Signal that a dependency is ready
await pd.publish('integration:ready:auth-service', { ready: true });

// Wait for a dependency (resolves immediately if already signaled)
await pd.integrationWait('auth-service');

// List sessions (optionally filter by phase)
const testing = await pd.listSessions({ phase: 'testing' });

// End session -- marks completed automatically
await pd.done({ sessionId, note: 'Auth merged, all tests green' });

// Abandon if you hit a blocker
await pd.updateSession(sessionId, { phase: 'abandoned' });

When to Use Each Phase

Phase selection guide

planning You're writing a design doc, deciding which files to change, or figuring out the approach. No code has been modified yet.
in_progress You've started making changes. Files are claimed, code is being written, the plan is executing.
testing Implementation is done. You're running the test suite, fixing failing tests, checking behavior with curl or a browser.
reviewing Code is working and tested. You've opened a PR, or you're waiting for a human or another agent to review your changes before merging.
completed Your PR is merged, or the task is definitively done and won't be revisited. Set automatically by pd done.
abandoned You've hit a blocker you can't resolve -- missing context, access issue, or the task needs a different approach. Use this so another agent can salvage your work with full context about where you stopped.
Keep phases updated as you go. A session stuck in planning while the agent is actually running tests is misleading to other agents. Update the phase when your work changes character -- it takes one command and makes the salvage system dramatically more useful.
